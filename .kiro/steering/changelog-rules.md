# Правила ведения CHANGELOG.md

## Общие принципы

1. Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.0.0/)
2. Проект придерживается [Semantic Versioning](https://semver.org/lang/ru/)
3. Все записи на русском языке

## Структура версии

Каждая версия должна содержать:

```markdown
## [X.Y.Z] - YYYY-MM-DD

### Added
- Новые функции и возможности

### Changed
- Изменения в существующем функционале

### Fixed
- Исправления багов

### Removed
- Удаленный функционал

### Deprecated
- Функционал, который скоро будет удален

### Security
- Исправления безопасности

### Технологии
- ТОЛЬКО новые технологии или значительные изменения версий
```

## ВАЖНО: Секция "Технологии"

### ❌ НЕПРАВИЛЬНО
Не нужно перечислять ВСЕ технологии проекта в каждой версии:

```markdown
### Технологии
- Go 1.25.5
- Node.js 25.2.0
- Next.js 16.0.8
- React 19.2.1
- TypeScript 5.x
- Tailwind CSS 4.x
- и т.д. (весь стек)
```

### ✅ ПРАВИЛЬНО
Указывать ТОЛЬКО то, что добавилось или изменилось:

```markdown
### Технологии
- Добавлено: DaData API для поиска организаций
- Добавлено: Yandex Maps API 2.1
- Обновлено: Next.js 15.0.0 → 16.0.8
```

Или если ничего не изменилось - вообще не включать секцию "Технологии".

### Примеры правильного использования

**Пример 1: Добавление новой технологии**
```markdown
### Технологии
- Добавлено: PostgreSQL 15 (миграция с SQLite)
- Добавлено: Redis для кеширования
```

**Пример 2: Обновление версии**
```markdown
### Технологии
- Обновлено: React 18.2.0 → 19.2.1
- Обновлено: TypeScript 4.9 → 5.0
```

**Пример 3: Новая возможность существующей технологии**
```markdown
### Технологии
- Next.js Dynamic Routes ([userId])
- Tailwind CSS Container Queries
```

**Пример 4: Ничего не изменилось**
```markdown
(секция "Технологии" отсутствует)
```

## Полный стек технологий

Полный список всех используемых технологий должен быть в **README.md**, а не в CHANGELOG.md.

CHANGELOG.md фиксирует только **изменения** в технологиях.

## Правила описания изменений

1. **Конкретность**: Описывайте что именно изменилось, а не общие фразы
2. **Пользовательская перспектива**: Пишите с точки зрения пользователя/разработчика
3. **Группировка**: Связанные изменения группируйте вместе
4. **Ссылки**: Если есть issue/PR, указывайте ссылку

### Примеры хороших описаний

✅ **Хорошо:**
- Добавлена возможность просмотра профилей других пользователей по адресу /id{номер}
- Исправлена ошибка 413 при поиске организаций (убран фильтр status)
- Кнопки редактирования профиля теперь показываются только владельцу

❌ **Плохо:**
- Обновлен код
- Исправлены баги
- Улучшена производительность

## Версионирование

- **Major (X.0.0)**: Несовместимые изменения API
- **Minor (0.X.0)**: Новый функционал, обратно совместимый
- **Patch (0.0.X)**: Исправления багов, обратно совместимые

## Unreleased секция

Всегда держите секцию `## [Unreleased]` вверху для будущих изменений.

## Дата релиза

Формат: `YYYY-MM-DD` (ISO 8601)

Пример: `2025-12-12`
